01 Context

Software estimates are usually wrong.

Plans assume the best case, are built on guesswork, and carry no buffer for the unknown. We built a tool that helps a team produce an estimate it can stand behind. These are the three failure patterns it is designed to counter.

Optimism bias

Plans assume the best case, with no room for the unknown.

Guesswork inputs

Estimates based on early talk, not the real scope of work.

No contingency

Risks get named, then quietly dropped to stay competitive.

COCOMO: size in, cost and contingency out.

How to use it

Enter the project size in KLOC and pick the project type. Effort, schedule and staffing update live, and the band shows the base estimate plus a contingency buffer.

02 Demonstration

Function Points: upload a CSV, read the size.

How to use it

A team drops in a CSV of its software components. The tool counts function points, applies the adjustment factor, and charts which components drive the size.

Validated

Reproduces the module test scenarios exactly: 142 UFP, 153.36 adjusted, 19.17 person-months.

Prototype first

We proved the estimation maths in a quick prototype before the full build, so we were never debugging the logic and the interface at the same time.

Kept it client-side

Everything runs in the browser, so there is no server to manage, each user works independently, and it is simple to host and demo live.

Held the scope

With only two days to build, we focused on cost estimation alone, delivering one complete, working tool rather than a wider one left unfinished.

05 Team reflection

How we worked, and what we learned.

Roles

Assigned to each member's strengths so we moved quickly from day one.

Decisions

Made by comparing options together, not defaulting to the first idea.

Challenges

Raised quickly as they came up, since we worked in the same room, with a timekeeper to keep us on track.

Learning

Applied lessons from working together before to run this project better.

We knew from working together before that we agreed too quickly and lost track of time, so this time we used a timekeeper and weighed options before committing. The progress that mattered most was in how we worked together.

What we would do differently

Set clearer hand-offs between building and testing, and agree the role split earlier, so we lose less time switching context partway through.
